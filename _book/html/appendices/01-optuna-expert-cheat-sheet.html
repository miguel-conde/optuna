Appendix A — Optuna Expert Cheat Sheet

Use this appendix as a compact reference for the patterns, APIs, and decision rules that matter most in day-to-day Optuna work. It is meant to complement the main chapters, not replace them. When you need the full reasoning behind a recommendation, follow the cross-links back into the corresponding module.

A.1 How To Use This Appendix

This appendix is organized around practical tasks:

  • defining an objective
  • choosing parameter types
  • selecting samplers and pruners
  • persisting and scaling studies
  • diagnosing search behavior
  • applying Optuna in machine-learning and production settings

If you only need a quick reminder, jump to One-Screen Summary. If you need a slightly more complete operational reference, scan the sections in order.

A.2 1. Basic Optimization Workflow

Minimal Optuna workflow:

import optuna


def objective(trial):
    x = trial.suggest_float("x", -10, 10)
    return (x - 2) ** 2


study = optuna.create_study(direction="minimize")
study.optimize(objective, n_trials=100)

print(study.best_params)
print(study.best_value)

Core workflow:

define objective
create study
run optimization
inspect results

For the deeper conceptual picture, see Basic Optuna Usage and Optuna Architecture.

A.3 2. Parameter Suggestion API

A.3.1 Continuous parameters

trial.suggest_float("lr", 1e-5, 1e-1)

A.3.2 Log-scale parameters

Use log scaling for multiplicative parameters such as learning rate, regularization, tolerances, and other quantities that span several orders of magnitude.

trial.suggest_float("lr", 1e-5, 1e-1, log=True)

A.3.3 Integer parameters

trial.suggest_int("depth", 2, 10)

A.3.4 Categorical parameters

trial.suggest_categorical("optimizer", ["adam", "sgd"])

For search-space design guidance, see Designing Search Spaces.

A.4 3. Conditional Search Spaces

Optuna supports dynamic, define-by-run spaces.

model = trial.suggest_categorical("model", ["rf", "svm"])

if model == "rf":
    depth = trial.suggest_int("depth", 3, 10)
else:
    c_value = trial.suggest_float("C", 1e-3, 10, log=True)

This creates hierarchical parameter spaces where later decisions depend on earlier choices.

That pattern becomes essential once model families, pipeline branches, or architectural decisions vary across trials.

A.5 4. Study Configuration

A.5.1 Default study

study = optuna.create_study(direction="minimize")

A.5.2 Custom sampler

study = optuna.create_study(
    sampler=optuna.samplers.TPESampler()
)

A.5.3 Persistent storage

study = optuna.create_study(
    storage="sqlite:///study.db",
    study_name="experiment",
    load_if_exists=True,
)

Persistent storage is the default choice once a study matters beyond one local session. See Distributed and Scalable Optimization and Production Workflows.

A.6 5. Common Samplers

Sampler Use case
TPESampler General hyperparameter tuning and mixed spaces
RandomSampler Baselines and debugging
CmaEsSampler Continuous optimization
GridSampler Very small discrete search spaces
NSGAIISampler Multi-objective optimization

Example:

sampler = optuna.samplers.CmaEsSampler()

study = optuna.create_study(
    sampler=sampler,
    direction="minimize",
)

The practical default for many problems remains TPE. Use other samplers when the geometry or objective structure clearly justifies it. See Samplers.

A.7 6. Pruning Pattern

Pruning is useful for iterative training loops.

for epoch in range(100):
    loss = train()
    trial.report(loss, epoch)

    if trial.should_prune():
        raise optuna.TrialPruned()

Create the study with a pruner:

study = optuna.create_study(
    pruner=optuna.pruners.MedianPruner()
)

Use pruning only when intermediate values are genuinely meaningful. See Pruning.

A.8 7. Best Trial Retrieval

After optimization:

study.best_value
study.best_params
study.best_trial

Those are the first values you usually inspect, but they are not the whole story. Diagnostics and trial history still matter.

A.9 8. Access All Trials

for trial in study.trials:
    print(trial.number)
    print(trial.params)
    print(trial.value)

The full trial list is useful for debugging, visualization, reporting, and understanding whether the study is converging or drifting.

A.12 11. Distributed Optimization

Shared database example:

study = optuna.create_study(
    storage="postgresql://user:pass@host/db",
    load_if_exists=True,
)

Run multiple workers:

python worker.py
python worker.py
python worker.py

All workers collaborate on the same study. For scaling trade-offs and storage choices, see Distributed and Scalable Optimization.

A.13 12. Parallel Execution

Single-machine parallelization:

This is usually the first scaling step before moving to shared-storage distributed execution.

Set seeds in Optuna:

Also control randomness in:

  • numpy
  • torch
  • tensorflow

Reproducibility also depends on data versioning, environment control, and explicit evaluation logic.

See Production Workflows.

A.17 16. Common Best Practices

A.17.1 Use log scales when scale matters

Good:

trial.suggest_float("lr", 1e-5, 1e-1, log=True)

A.17.2 Avoid huge ranges without justification

Bad:

learning rate: [1e-6, 10]

A.17.3 Use cross-validation when appropriate

cross_val_score(model, X, y, cv=5)

A.17.4 Use pipelines to avoid leakage

Pipeline([
    ("scaler", StandardScaler()),
    ("model", RandomForest()),
])

A.17.5 Use pruning for iterative workloads

Pruning is often valuable in deep learning, boosting, and long-running iterative objectives.

A.18 17. Common Optimization Pattern

Recommended workflow:

1 design search space
2 run 100-300 trials
3 visualize results
4 refine search space
5 run second optimization

Optimization is usually iterative. The first study often teaches you how to design the second one better.

A.19 18. Debugging Optimization

Signs of a weak search space or unstable objective:

Symptom Likely cause
no convergence space too large
flat slice plot parameter irrelevant or poorly explored
boundary solutions range too narrow or badly centered
unstable results noisy objective

A.20 19. Typical Trial Budgets

Problem Trials
small models 50-100
ML pipelines 100-300
deep learning 200-1000
simulation optimization 500+

These are practical heuristics, not rules. Objective cost and noise matter as much as problem type.

A.21 20. Mental Model Of Optuna

Optuna optimization loop:

Sampler proposes parameters
        ↓
Objective evaluates
        ↓
Result stored
        ↓
Sampler updates strategy
        ↓
Next trial

This continues until one of the stopping criteria is met:

  • n_trials
  • timeout
  • manual stop

A.22 One-Screen Summary

trial.suggest_* -> define search space
objective()     -> compute metric
study.optimize  -> run trials
sampler         -> choose parameters
pruner          -> stop bad trials
visualization   -> analyze results
storage         -> persist experiments

A.23 Final Note

This cheat sheet is a quick reference, not a substitute for workflow judgment. The strongest Optuna users are not the ones who memorize the most methods. They are the ones who design the objective carefully, shape the search space thoughtfully, and interpret optimization results with discipline.
